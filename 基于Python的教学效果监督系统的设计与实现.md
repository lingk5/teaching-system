# 基于Python的教学效果监督系统的设计与实现

## 摘要
随着高校教学活动逐步数字化，课堂考勤、作业提交、随堂测验和课堂互动等过程性数据不断积累。相比期末成绩这类结果性数据，过程性数据更能反映学生在学习过程中的真实状态，但在实际教学中，这些数据往往分散在不同表格、不同平台和不同教师手中，统计方式不统一、处理效率较低，导致教师难以及时发现学业风险学生，也难以形成持续性的干预依据。针对这一问题，本文设计并实现了一套基于 Python 的教学效果监督系统。

系统采用前后端分离架构。后端以 Flask 为核心框架，结合 SQLAlchemy 完成用户、课程、班级、学生、过程数据和预警记录等业务对象建模；数据库当前默认使用 MySQL，并通过 ORM 保持后续迁移的灵活性；在数据处理环节，引入 Pandas 与 OpenPyXL 完成 Excel 数据导入、字段清洗、格式校验和错误反馈；前端采用原生 HTML、CSS 与 JavaScript 模块化开发，结合 Bootstrap 构建页面布局，结合 Chart.js 实现学情信息的图表展示。系统围绕教学监督场景实现了登录认证、角色权限控制、管理员用户管理、课程与学生管理、多类型教学数据导入、综合预警生成与处理、基础统计与报表导出等功能。

在预警机制方面，系统基于出勤、作业、测验、期末和互动五类指标构建综合评分模型，并结合等级划分规则生成黄色、橙色和红色预警，辅助教师快速识别重点关注对象。与只依赖单一阈值判断的做法相比，当前实现更强调多维度汇总、覆盖率约束和结果解释，能够为教师提供较为直观的原因说明和处理建议。测试结果表明，系统当前版本已经完成主流程闭环、三角色权限分离和期末成绩独立建模，能够满足毕业设计演示与论文论证的需要。

**关键词**：教学监督；Python；Flask；数据导入；学情预警；角色权限控制

## ABSTRACT
With the digitalisation of higher education, process data such as attendance, homework, quizzes, and classroom interaction are generated continuously. Compared with final scores, process data can better reflect students' learning status during the semester. However, in actual teaching practice, these data are often scattered across spreadsheets and platforms, which leads to inconsistent statistical standards and low management efficiency. As a result, teachers find it difficult to identify at-risk students in time and carry out continuous intervention. To address this problem, this thesis designs and implements a teaching effectiveness monitoring system based on Python.

The system adopts a front-end and back-end separated architecture. The back end is built with Flask and SQLAlchemy to model users, courses, classes, students, process data, and warning records. MySQL is used as the default database in the current version, while the ORM layer preserves migration flexibility. Pandas and OpenPyXL are introduced to support Excel-based data import, field cleaning, format validation, and error feedback. The front end is developed with native HTML, CSS, and modular JavaScript, combined with Bootstrap for page layout and Chart.js for data visualisation. The system implements login authentication, role-based access control, administrator user management, course and student management, multi-type data import, comprehensive warning generation and handling, as well as basic statistics and report export.

For the warning mechanism, the system integrates attendance, homework, quiz, final examination, and interaction indicators into a comprehensive scoring model, and classifies students into yellow, orange, and red warning levels. Compared with single-threshold judgement, the current implementation emphasises multidimensional aggregation, coverage constraints, and interpretable results, which helps teachers understand the main reasons behind risks and take timely action. Current verification results show that the system has completed the core workflow, role separation among administrator, teacher, and assistant, and independent final-score modelling, making it suitable for graduation project demonstration and thesis discussion.

**Key words**: teaching supervision; Python; Flask; data import; early warning; role-based access control

## 第一章 绪论

### 1.1 研究背景
近年来，高校教学活动逐渐从单纯依赖结果评价转向更加重视过程评价。尤其是在混合式教学和信息化教学环境中，学生在整个学期中会持续产生大量过程性数据，例如签到记录、作业得分、随堂测验成绩、课堂互动次数和阶段性预警信息。这些数据如果能够被及时采集、清洗、分析并形成反馈，就可以帮助教师在期中阶段发现问题，而不是等到期末成绩出来后再进行被动补救。

但在实际教学管理中，过程数据常常分散在不同渠道之中：有的存在教师自己维护的 Excel 表格中，有的来自在线学习平台，有的来自课堂互动工具，还有一些依赖人工汇总。由于采集口径不统一、格式不规范、缺少持续的数据整理机制，教师往往需要投入大量时间进行重复统计，最终得到的结果也难以形成统一的学情视图。这种状况不仅影响教学效率，也削弱了过程性评价在教学改进中的实际作用。

在这一背景下，设计一套面向教学监督场景的轻量化系统具有现实意义。该系统不需要完全替代学校现有教务平台，而是作为过程性数据的聚合、分析和预警工具，帮助管理员维护基础数据，帮助教师完成过程信息录入和查看，也为助教提供只读辅助能力，从而形成更清晰的教学监督链路。

### 1.2 研究意义
本课题的研究意义主要体现在三个方面。

第一，具有明显的教学管理意义。系统将课程、班级、学生、考勤、作业、测验、期末成绩、互动和预警等信息集中到统一平台中，有助于减少教师在数据整理上的重复劳动，提高教学数据的可追踪性和可查询性。

第二，具有过程干预意义。与传统只关注期末成绩的做法相比，本系统强调过程性数据的持续汇总与预警生成。教师可以在学期中根据学生的综合得分和预警等级进行重点关注，从而把教学干预前移。

第三，具有工程实践意义。系统采用 Flask、SQLAlchemy、Pandas、MySQL、Bootstrap 和原生 JavaScript 等较为成熟且成本可控的技术栈，实现了一套结构清晰、便于部署和扩展的 Web 系统，为类似教学辅助平台的开发提供了可参考的实现路径。

### 1.3 国内外研究现状
结合已查阅的相关文献，现有研究大致可以分为三类。第一类聚焦 Python 课程教学改革与教学模式优化。冀全朋等从项目教学角度对课程内容分层组织，强调通过基础类、进阶类和开发类项目提升学生编程兴趣与程序设计能力[8]；夏晓帅从计算思维培养视角构建项目式教学模式，指出问题驱动和成果交流有助于提升算法设计与问题解决能力[9]；张进、吴婧妍分别从学习共同体和学习投入角度对初中、高中 Python 教学流程进行重构，强调真实合作学习、过程反馈和“导—学—练—评”闭环对课堂效果的促进作用[10][11]。此外，唐茂叶、彭思慧等研究进一步把 AI 双师、线上线下融合以及生成式人工智能引入编程教学场景，说明智能技术能够在情境创设、资源生成、个别答疑和过程评价中发挥支撑作用，但同时也带来了技术依赖、学业诚信和平台适配等问题[12][13]。黄健、唐颖等则从知识图谱和教学平台建设角度推动 Python 课程资源组织与计算思维培养[14][15]。这些研究说明，当前 Python 教学研究已经从单纯语法讲授逐步转向项目实践、协作学习、过程投入和智能支持并重的方向。

第二类研究主要围绕教学管理系统和在线学习平台的设计与实现展开。雷晓薇基于 Django 构建了教学管理系统，将信息发布、交流答疑、课程管理和成绩查询等功能整合到统一平台中[16]；钟如秀围绕在线视频学习场景设计在线学习系统，强调课程资源、评论互动和个人中心等模块[17]；吕丽君、李娜分别从课堂管理和成绩管理出发，采用 Python 实现座位分配、考勤、随机提问、成绩录入、查询排序和统计分析等功能[18][19]。这类研究为教学信息化平台的模块设计、数据库组织和界面实现提供了较多经验，但总体上更偏向事务管理或单场景应用，对多源过程性数据的综合汇聚、跨模块联动分析和风险预警支持相对有限。

第三类研究开始关注教学质量监测、学习风险识别和数据驱动干预。Wang 提出的 AI 学生管理系统研究强调利用混合智能模型对学生成绩表现进行持续监测和干预[20]；Trabelsi 等通过深度学习识别课堂行为和注意力状态，为课堂参与度评价提供了新的技术路径[21]；Shuyi Wang 从 OBE 理念出发，指出传统教学质量监测中存在评价标准单一、过程评价缺失和反馈滞后等问题，主张建立面向结果与过程协同的质量监测体系[22]。这些研究说明，教学监督系统的发展趋势已经从静态管理转向过程评价、行为识别和风险预警，但很多方案依赖较重的平台生态或复杂数据采集条件，在中小规模课程场景中仍存在部署和适配门槛。

总体来看，现有研究为本课题提供了三方面启示：其一，教学监督系统不能仅停留在课程信息维护层面，而应结合过程性数据形成持续反馈；其二，平台设计既要考虑教师日常使用习惯，也要兼顾过程评价与预警解释；其三，系统不必一开始就构建过重的综合平台，而可以围绕具体课程监督场景，从轻量导入、角色协同和可解释预警切入。因此，本文选择设计一套面向课程级应用的教学效果监督系统，在已有教学平台和教学模式研究基础上，进一步强化过程数据汇聚、角色权限边界和预警闭环处理能力。

## 第二章 相关平台及关键技术

### 2.1 Python 与 Flask 框架
Python 具有语法清晰、生态丰富、开发效率较高等特点，尤其适合数据处理与 Web 服务结合的场景。本系统以后端接口服务和数据处理为核心，因此选择 Python 作为主要开发语言。

在 Web 框架方面，系统采用 Flask 作为后端基础框架。Flask 本身结构轻量、路由机制清晰，适合在毕业设计场景下快速构建 RESTful API。当前系统通过蓝图机制将接口划分为认证、课程管理、数据导入、学情分析、预警处理和数据导出等模块，既降低了代码耦合度，也便于后续迭代。

在身份认证方面，系统采用基于 JWT 的无状态鉴权机制。登录成功后，后端会将用户标识和角色信息写入令牌，并在后续请求中完成身份校验。相比传统服务端会话方案，该方式更适合前后端分离的应用结构。

### 2.2 SQLAlchemy、MySQL 与数据建模
数据库是系统稳定运行的基础。当前项目默认使用 MySQL 作为关系型数据库，主要原因在于其部署成熟、使用广泛，并能够满足教学监督系统对课程、学生和过程数据的持久化需求。

为了降低业务代码与底层 SQL 的耦合，系统使用 SQLAlchemy 作为 ORM 工具，将用户、课程、班级、学生、考勤、作业、测验、期末成绩、互动和预警等对象映射为数据库模型。通过这种方式，开发者可以在保持数据结构清晰的同时，提高查询、维护和扩展效率。

当前系统中，用户与课程之间是一对多关系，课程与班级之间是一对多关系，班级与学生之间是一对多关系。出勤、作业、测验、期末成绩、互动和预警等过程数据既与学生关联，也保留课程维度，用于同时定位“谁产生了数据”和“数据属于哪门课程”。其中预警记录除关联学生和课程外，还可记录处理状态与处理人信息。该建模方式能够较好支撑当前业务流程，也便于后续开展按课程、按学生两个维度的统计分析。

### 2.3 Pandas、OpenPyXL 与数据导入
教学监督系统的一个典型场景是教师批量导入 Excel 数据。由于教师实际维护的表格格式常常存在空行、空格、字段缺失、状态值不规范等问题，系统必须具备较强的数据容错能力。

Pandas 在表格数据读取、缺失值处理和字段标准化方面具有较高效率，OpenPyXL 则适合处理 Excel 文件结构和导出样式。本系统在导入环节结合这两类工具，对学生名单、考勤、作业、测验、期末成绩和互动数据进行解析、校验和落库，并将成功条数、失败条数和错误原因返回给前端。这一机制是系统实用性的重要基础。

### 2.4 前端技术与可视化实现
当前前端采用原生 HTML、CSS 和 JavaScript 模块化开发，没有引入 Vue、React 等重型框架。这种实现方式虽然在组件化能力上不如大型前端框架完整，但更符合当前项目轻量、直接和易部署的目标。

页面层面使用 Bootstrap 完成布局、表单和表格样式，以减少基础界面开发成本；交互层面通过 `request.js` 封装网络请求，通过 `auth.js` 统一处理登录状态、角色主页跳转、页面访问控制和基于能力的控件显示；图表层面使用 Chart.js 绘制预警分布、趋势图和雷达图等可视化内容。当前版本的仪表盘和分析页已经移除模拟数据兜底，当缺少真实教学数据时直接显示空状态，这使系统展示结果与后端真实数据保持一致。

## 第三章 系统需求分析

### 3.1 可行性分析
#### 3.1.1 技术可行性
从技术角度看，系统所采用的 Flask、SQLAlchemy、MySQL、Pandas、Bootstrap 和 Chart.js 都属于成熟技术，社区资料丰富、实现路径清晰，适合毕业设计项目的开发与维护。Flask 作为轻量级 Web 框架，能够较好满足本系统接口清晰、模块边界明确的开发需求；SQLAlchemy 负责数据库对象映射，可以降低底层 SQL 编写成本，方便后续维护；Pandas 与 OpenPyXL 则适合处理教学场景下大量 Excel 数据导入与导出任务；Bootstrap 和 Chart.js 为前端页面展示与图表分析提供了较直接的实现方式。

从当前项目实现情况来看，系统已经完成登录认证、角色权限划分、管理员用户管理、课程和学生信息维护、教学过程数据导入、预警计算与处理、统计展示和基础导出等主要模块，说明整体技术路线并非停留在概念层面，而是具备了实际落地基础。此外，后端已经形成蓝图、模型、服务层和工具函数相对分离的代码结构，前端也建立了统一请求模块与权限控制模块，这使得系统在毕业设计规模下具备较好的实现可行性和后续扩展空间。

#### 3.1.2 经济可行性
系统采用的核心框架和工具大多为开源方案，不需要额外的商业授权成本。对于教学演示场景而言，一台普通开发环境即可完成运行、展示和简单测试，部署成本较低，符合毕业设计项目的资源条件。在开发过程中，主要投入集中于个人时间成本和测试整理成本，而不需要额外采购商用数据库、可视化平台或外部接口服务，因此整体投入相对可控。

对于高校课程教学监督这一应用场景而言，系统重点解决的是教学数据整合、过程监督和风险提示问题，使用对象主要是管理员、教师和助教等校内角色，不涉及大规模商业化推广阶段所需的高额服务器与运维支出。因此，在毕业设计阶段采用本地部署或局域网演示方式即可满足展示需求。从资源占用、硬件要求和开发成本三个角度综合判断，本系统具有较好的经济可行性。

#### 3.1.3 业务可行性
从教学业务出发，课程过程监督确实存在基础数据分散、统计负担重和干预不及时的问题。在传统教学管理方式中，教师往往需要分别在表格、纸质记录或多个平台中维护学生名单、考勤结果、作业成绩和课堂互动情况，既增加了重复劳动，也导致数据难以形成统一视图。当教师需要快速识别学习状态异常的学生时，往往只能凭经验进行判断，缺乏较为系统的数据支撑。

当前系统以课程、班级和学生为对象，围绕教师日常管理流程设计，能够覆盖登录、基础信息维护、批量导入、风险预警、处理登记和结果导出等关键环节，与实际教学管理习惯具有较高贴合度。同时，系统中引入管理员、教师和助教三类角色，可以较好映射现实中的工作分工关系。管理员负责账号与全局维护，教师负责课程运行与预警处理，助教负责辅助查看和支持，这种角色划分具备明确的业务逻辑基础。因此，从实际应用需求和使用流程来看，本系统具备较强的业务可行性。

### 3.2 角色与功能需求分析
当前系统面向三类角色开展设计，分别为管理员、教师和助教。

#### 3.2.1 管理员需求
管理员负责系统级基础维护，主要需求包括：

1. 登录系统并进入管理员控制台。
2. 创建、查看、修改教师和助教账号。
3. 查看课程、学生和整体统计信息。
4. 参与课程维护、数据导入和预警查看。

### 3.3 功能需求分析
结合当前仓库实现，系统功能需求可概括为以下几个部分。需要说明的是，需求分析并不仅仅是对现有功能列表进行罗列，更重要的是解释系统为什么需要这些功能、这些功能之间如何衔接以及它们分别服务于哪些教学管理目标。对于教学效果监督系统而言，角色权限、基础数据、过程数据、预警分析与结果导出共同构成了一个相互关联的业务闭环。

#### 3.3.1 登录认证与权限控制
系统需要支持登录、身份验证、角色识别和页面访问控制。不同角色登录后应进入不同首页，并看到与自身职责相符的功能入口。管理员需要拥有独立用户管理页面，教师不应拥有用户管理能力，助教不应拥有课程创建和导入能力。这一需求的核心并不只是“能否登录”，而是系统能否在登录之后继续维持稳定且清晰的权限边界。

在教学管理场景中，不同角色对同一数据的使用方式并不一致。管理员更关注用户维护和全局监督，教师更关注课程运行和学生状态，助教则主要承担查看与辅助工作。因此，系统必须能够在页面访问、按钮显示和关键接口调用三个层面同时识别角色身份，避免出现界面可见但接口越权，或者接口可控但页面混乱的情况。只有在认证和授权逻辑稳定的前提下，后续课程维护、数据导入和预警处理等业务模块才能可靠运行。

#### 3.3.2 基础数据管理
系统需要支持课程、班级和学生信息维护，能够建立课程到班级、班级到学生的层级关系。管理员和教师需要具备相应的数据维护能力，助教仅保留查看权限。基础数据管理是整个系统的结构骨架，如果课程、班级和学生之间的关系不能被清晰建立，那么后续的考勤、作业、测验、互动和预警数据将无法准确归属。

从业务流程上看，教师通常先确定课程信息，再建立课程下的班级结构，随后维护班级中的学生名单。系统因此需要具备课程创建、课程列表查看、班级信息维护、学生基本信息管理等能力，并保持不同层级对象之间的一致性。同时，这些基础数据还需要为导入模块提供参照，例如导入考勤记录时必须能够定位到对应课程与学生。由此可见，基础数据管理既是单独功能模块，也是支撑其他功能正常运行的前提条件。

#### 3.3.3 教学数据导入
系统需要支持通过 Excel 批量导入学生名单、考勤、作业、测验、期末成绩和互动数据，并在导入失败时返回具体错误原因，帮助教师快速修正数据源。在真实教学环境中，教师往往已经习惯使用电子表格记录课堂过程信息，因此系统若要求逐条录入数据，不仅效率低，而且会显著提高使用门槛。支持批量导入是提升系统实用性的关键需求之一。

然而，教学数据导入并不是简单地把 Excel 内容写入数据库。教师提交的原始表格往往存在字段命名不统一、空值较多、状态表达不规范、学生编号缺失或重复等问题，因此系统还需要在导入过程中完成模板识别、字段映射、空值处理、合法性校验和错误汇总。只有当系统能够准确区分可写入数据与异常数据，并将错误原因明确反馈给使用者时，导入模块才真正具备业务价值。

#### 3.3.4 预警与处理
系统需要对学生的过程性数据进行计算，并生成可视化的风险提示与处理建议。教师和管理员应能够查看并处理预警，助教仅能查看。预警模块是本系统区别于普通信息管理系统的核心功能，也是教学效果监督这一主题的重要体现。

从业务目标上看，系统并不仅仅满足于记录出勤、作业和测验结果，而是希望通过整合这些过程性数据，帮助教师更早发现潜在风险学生。因此，系统需要能够根据考勤、作业、测验和互动等指标计算综合分值，并依据预设阈值划分不同风险等级。同时，系统还应支持记录预警原因、处理建议和后续处理状态，使预警结果从单纯“发现问题”进一步延伸到“跟踪问题”。这样才能使系统形成从数据采集、风险识别到人工干预的完整链路。

#### 3.3.5 数据展示与导出
系统需要提供基础统计与报表导出能力，方便教师查看课程情况，并将学生名单、成绩、考勤和预警数据导出为 Excel 文件。对于教学管理者而言，系统的价值不仅在于完成数据录入和预警计算，更在于是否能够将这些结果以便于阅读和后续使用的方式输出出来。

一方面，系统需要提供基础统计展示能力，例如学生数量、课程数量、预警人数以及部分图表分析结果，帮助教师快速了解整体教学运行状态；另一方面，系统还需要支持按业务场景导出结果，将系统内的数据转换为可留档、可汇报和可继续加工的 Excel 报表。特别是在教学检查、班级汇总和过程考核等情境下，导出功能能够明显提高系统的实用价值，因此应被视为完整业务闭环的重要组成部分。

### 3.4 非功能需求分析
#### 3.4.1 易用性
系统面向教师和教学管理人员使用，因此界面应尽量简洁，操作路径应保持直接。当前版本通过固定侧边栏、统一表格和模态框表单组织主要功能，降低了学习成本。由于系统的主要使用者并不一定具备专业的信息技术背景，因此界面设计必须强调清晰、稳定和低认知负担，不能将复杂操作隐藏在过深的层级结构中。

从实际使用角度看，易用性主要体现在三个方面。第一，角色登录后应能快速进入与自身职责匹配的页面，而不必在无关模块之间反复切换；第二，表格、按钮、弹窗和状态提示应尽量保持统一风格，使教师能够通过重复使用形成操作习惯；第三，导入失败、权限不足和数据为空等常见场景必须给出明确反馈，避免用户在不清楚原因的情况下重复尝试。只有在这些细节上保持友好，系统才能真正被教学场景接受。

#### 3.4.2 可维护性
后端通过蓝图、服务层和工具函数拆分模块，前端通过脚本模块和页面职责分离来降低耦合，能够提升后续维护效率。数据库模型与接口文件职责相对明确，也有利于版本迭代。对于毕业设计项目而言，可维护性不仅关系到当前开发效率，也关系到后续修改论文、补充演示和扩展功能时是否容易定位问题。

本系统当前已经形成相对明确的模块边界，例如认证、课程管理、数据导入、预警处理和导出等逻辑分别由不同文件承担，角色权限判断也开始由统一能力矩阵进行收口。这种设计方式有助于减少“一个文件承担过多职责”的问题，提高代码可读性与修改稳定性。与此同时，数据库实体关系较为清晰，前后端接口名称与页面功能之间也具有一定对应关系，这些都为后续继续优化系统提供了较好的维护基础。

#### 3.4.3 安全性
系统当前采用 JWT 鉴权与角色能力矩阵控制关键操作，在用户管理、课程创建、数据导入和预警处理等环节增加了权限校验，能够为基础教学场景提供必要的安全边界。由于系统中涉及教师账号、课程信息、学生名单、成绩与预警结果等敏感数据，因此安全性需求虽然不必达到大型商业平台级别，但也不能被忽视。

具体而言，系统至少需要满足三个方面的安全要求。第一，未登录用户不得访问受保护页面和接口，令牌失效后应能够自动跳转登录页；第二，不同角色只能执行授权范围内的操作，避免助教获取管理员或教师权限；第三，数据导入、预警处理和用户管理等关键功能必须同时在前端和后端进行权限限制，防止仅通过隐藏按钮实现“表面安全”。在毕业设计场景中，这种基础安全边界已经能够满足系统演示与应用论证需要。

## 第四章 系统设计

### 4.1 总体架构设计
系统采用前后端分离架构，整体可分为表现层、接口层、业务层和数据层四个部分。

表现层由浏览器中的前端页面构成，包括登录页、仪表盘、课程管理、学生管理、用户管理、数据导入、学情分析和预警中心等页面。这些页面通过统一的请求模块访问后端接口，并根据角色和能力动态控制菜单与按钮展示。

接口层由 Flask 蓝图构成，主要包括认证模块、课程模块、数据导入模块、分析模块、预警模块和导出模块。接口层负责接收请求、校验参数、调用业务逻辑，并向前端返回 JSON 数据或导出文件。

业务层主要包含权限判断、预警引擎、数据导入辅助函数和统计逻辑。预警引擎会根据学生最近一段时间的出勤、作业、测验、期末成绩和互动情况计算综合得分，并输出等级、原因和建议。

### 4.3 数据库设计
#### 4.3.1 核心实体设计
结合当前实现，系统涉及以下核心实体：

1. 用户实体，用于存储管理员、教师和助教账号。
2. 课程实体，用于存储课程基础信息并与教师关联。
3. 班级实体，用于表示课程下的班级结构。
4. 学生实体，用于表示学生基本资料并与班级关联。
5. 出勤、作业、测验、期末成绩和互动实体，用于存储教学过程与结果性数据。
6. 预警实体，用于存储学生预警等级、原因说明、建议和处理状态。

#### 4.3.2 实体关系说明
当前数据库关系如下：

1. 一个用户可以关联多门课程。
2. 一门课程可以包含多个班级。
3. 一个班级可以包含多个学生。
4. 出勤、作业、测验、期末成绩、互动等教学记录同时关联学生和课程，用于保留个体行为与课程场景两个维度。
5. 预警记录同时关联学生和课程，并可进一步记录处理状态和处理人。

## 第五章 系统实现

### 5.1 开发环境与项目组织
在系统实现阶段，开发环境的选择首先服务于“能够稳定完成主流程开发与演示”这一目标。当前系统采用 Python 3 作为主要开发语言，后端依赖 Flask、Flask-JWT-Extended、Flask-SQLAlchemy、Pandas、OpenPyXL 和 PyMySQL 等库，前端则采用浏览器静态页面模式，主要依赖 Bootstrap 与 Chart.js。这样的技术组合并不追求复杂框架堆叠，而是更强调结构清晰、部署直接和便于论文展示。

从应用初始化方式看，系统通过 `create_app` 工厂函数完成数据库、JWT 和跨域配置，并在启动阶段创建默认演示账号 `admin`、`teacher` 和 `assistant`，用于本地调试和功能验证。这样的组织方式有助于将应用初始化逻辑与业务逻辑分离，降低配置分散所带来的维护成本。同时，默认演示账号的保留也使系统在答辩和论文展示阶段能够快速进入核心业务流程。

在项目组织上，后端路由位于 `backend/app/routes`，模型位于 `backend/app/models`，预警与权重逻辑位于 `backend/app/services`；前端页面位于 `frontend/src/pages`，公共脚本位于 `frontend/src/js`。这种目录划分并不是简单的文件归类，而是试图让“接口处理”“数据建模”“业务逻辑”“页面展示”四类职责保持相对清晰的边界。对于毕业设计项目而言，这种结构既有助于快速定位问题，也使论文中各模块的描述能够与代码组织一一对应，从而提升论述的一致性与可验证性。

### 5.2 登录认证与角色权限实现
登录认证模块的实现目标，是建立系统最基础的访问入口，并为后续角色权限控制提供身份依据。系统在登录阶段由前端登录页收集用户名和密码，后端完成账号校验后生成携带用户标识与角色信息的 JWT 令牌。前端在接收到认证结果后，将令牌与必要的用户信息写入本地存储，并在后续页面访问时自动附带认证信息，从而形成较完整的登录链路。

如果只实现登录而不继续约束权限，系统仍然会出现角色边界模糊的问题。因此，角色权限控制由前端和后端共同完成。前端 `auth.js` 维护三类角色的首页映射：管理员默认进入 `users.html`，教师进入 `dashboard.html`，助教进入 `analytics.html`。同时，前端还根据能力集合控制“用户管理”“创建课程”“数据导入”“生成预警”“处理预警”等入口的显示，以减少无权限用户看到不应操作内容的情况。

仅依赖前端隐藏按钮并不能真正构成安全边界，因此后端 `permissions.py` 又对关键接口进行统一能力校验。这样一来，即使用户绕过页面直接发起请求，也会因为角色不匹配而被拒绝。经过这一轮实现与修正，系统已经摆脱了管理员、教师和助教侧边栏完全相同、权限边界不清的问题。管理员拥有完整能力，教师不具备用户管理能力，助教不具备课程创建、数据导入和预警处理能力。以反思性的角度来看，这一模块的价值不只是“能够登录”，而是在于它真正收口了系统早期最明显的权限割裂问题。

### 5.3 管理员用户管理与基础数据维护实现
管理员用户管理模块的实现目标，是让管理员和教师在界面职责上真正分离，而不是继续共用一套侧边栏和操作页面。为满足这一要求，系统恢复并实现了管理员专用的 `users.html` 页面。该页面仅允许管理员访问，并对接后端 `/api/auth/users` 相关接口，实现用户列表查看、创建用户和编辑用户等操作。通过这一实现，管理员能够在系统内部直接完成教师和助教账号维护，而不再依赖外部手工处理。

从实现过程来看，用户管理页并不只是一个独立页面，而是权限矩阵与后端认证接口协同工作的结果。管理员登录后会直接进入用户管理入口，并在页面中完成用户查询、账号新增、角色区分和状态修改等操作。这使系统的“谁可以进入系统、以什么身份进入系统”第一次被纳入同一管理界面，也使管理员角色具备了更清晰的系统级维护意义。

在课程与学生数据维护方面，后端 `courses.py` 负责课程、班级和学生的查询与增删改查逻辑，前端分别通过课程管理页和学生管理页展示层级信息。课程页面支持查看课程下的班级情况和预警人数，学生页面支持查看学生列表及基础画像信息。更重要的是，这些基础数据维护能力并不是孤立存在的，它们直接决定后续导入、预警和导出模块是否能够建立正确的数据归属关系。因此，从实现效果上看，该模块构成了教学监督所需的基础数据闭环，也是后续业务链路得以成立的前提。

### 5.4 数据导入实现
数据导入模块是本系统的重要实现内容之一，也是系统贴近真实教学场景的关键体现。由于教师在日常教学中通常使用 Excel 表格记录学生名单、考勤状态、作业成绩、测验结果和课堂互动情况，因此系统必须能够适配这类现成数据，而不是要求教师重新逐条录入。基于这一目标，后端 `data.py` 支持学生、考勤、作业、测验和互动等多类数据导入，并根据不同类型调用对应导入器完成解析和落库。

从实现方式上看，导入过程并不是简单读取表格后直接写入数据库，而是包含了字段验证、空值处理、状态转换、重复记录判断和错误信息收集等步骤。系统会先根据用户选择的数据类型识别表格模板，再利用 Pandas 对数据进行读取和清洗，最后在满足合法性条件时执行写入，在异常情况下保留错误原因。这样设计的目的，是让系统在面对真实表格时具备一定容错能力，而不是只接受理想化输入。

前端数据导入页面为教师和管理员提供模板下载、文件上传和结果反馈功能。导入结束后，系统会返回成功数量、失败数量及具体错误原因，帮助用户快速定位问题。以反思的角度来看，导入模块的价值并不在于“能上传文件”这一表面功能，而在于它将教师已有的数据工作方式与系统内部数据结构有效衔接起来，减少了系统落地时最常见的使用阻力。

### 5.5 预警生成、处理与可视化实现
预警模块是本系统中最能体现“教学效果监督”主题的实现内容。与普通教学管理系统只保存成绩或考勤结果不同，本系统尝试通过过程性数据的综合分析，对学生学习状态进行更早期的识别和提示。预警生成逻辑由 `warning_engine.py` 完成，引擎当前选取出勤、作业、测验和互动等指标，分别换算为百分制后参与综合评分，再通过统一权重配置生成综合结果。

从实现逻辑上看，系统并不直接以单项成绩高低判断学生风险，而是先按课程维度汇总学生最近一段时间的教学数据，再计算综合分，并根据阈值映射为黄色、橙色和红色三个等级。对于达到预警条件的学生，系统会进一步生成原因说明和建议文本，并将预警记录写入数据库。这种设计使预警结果不再是抽象分数，而是能够在页面上形成“等级 + 原因 + 建议”的可解释输出。

预警中心页面负责展示预警信息及处理状态。教师和管理员可以在该页面查看学生风险等级、处理建议和处理进展，并记录预警处理结果；助教仅保留查看权限，不允许进行处理操作。这使预警模块不再停留在“发现问题”层面，而是进一步延伸到了“处理问题”和“记录问题处理过程”的环节，形成了较为完整的业务闭环。

在数据可视化方面，系统当前提供仪表盘和学情分析页面，用于展示统计信息和图表内容。预警分布、学生信息展示和图表数据均已接入真实接口；当某课程缺少真实教学数据时，页面直接展示空状态而不再使用模拟数据兜底。这样的实现虽然仍属于基础分析层面，但它体现了一个重要的反思性取向：系统应当如实反映数据状态，而不是用虚拟结果掩盖真实缺口。

### 5.6 数据导出实现
数据导出模块的实现目标，是让系统内形成的数据结果能够继续服务于教学管理实际，而不是停留在页面展示层面。当前系统已经实现基础 Excel 导出能力，后端 `export.py` 支持导出学生名单、成绩报表、考勤统计和预警报告，并通过 OpenPyXL 为表头和边框添加基础样式。这样一来，教师不仅可以在系统中查看数据，也可以将结果用于留档、汇报或进一步加工分析。

从实现方式上看，导出模块本质上是对查询结果的再组织过程。系统先依据导出类型生成相应数据集，再将其写入工作簿中形成结构化报表。相比在页面上逐项复制数据，导出能力更适合教学管理中的实际使用情境，例如课程归档、班级汇总、成绩复核和预警留痕等。因此，该模块虽然技术复杂度不高，但在系统完整性上具有重要意义。

需要说明的是，当前实现侧重 Excel 报表输出，并未实现论文二稿中曾经提到的 PDF 学情报告能力。因此在论文正文中应以 Excel 导出为准。这种表述上的收缩并不是功能退步，而是对系统当前真实完成度的如实反映。对于毕业论文而言，如实呈现“已经完成什么、尚未完成什么”比单纯追求功能描述丰富更有说服力。

## 第六章 系统测试

### 6.1 测试环境与测试方法
为了验证当前版本是否具备演示与论文论证基础，系统采用了自动化回归验证与源码级检查相结合的方式进行测试。后端使用 Python `unittest` 对角色权限和页面回归进行验证，前端使用 `node:test` 对角色矩阵和控件裁剪逻辑进行验证，同时结合 `compileall` 和补丁检查命令确认代码基础状态。

本次测试围绕以下几个重点展开：

1. 管理员、教师、助教三角色的权限是否已经分离。
2. 管理员用户管理界面与接口是否可用。
3. 助教是否仍然可以进行不应拥有的高权限操作。
4. 前端页面访问与按钮显示是否与后端能力控制保持一致。
5. 当前代码是否能够通过基础自动化回归和编译检查。

### 6.2 测试结果
根据当前代码基线的实际验证结果，系统已经建立自动化回归测试集。后端共完成 32 条测试，前端共完成 7 条测试，此外通过了 Python 编译检查和补丁格式检查。

在角色权限方面，测试确认了以下结论：

1. 管理员登录后默认进入用户管理页面。
2. 管理员可以访问用户列表接口。
3. 教师访问用户管理接口会被拒绝。
4. 助教无法导入课程数据。
5. 助教无法修改学生信息。
6. 助教无法处理预警。
7. 管理员页面文件已存在且已接真实接口，不是空白演示页。

在前端回归方面，测试确认 `auth.js` 中的角色主页、页面访问矩阵和能力裁剪逻辑能够按预期工作，能够正确隐藏助教不应看到的高权限控件，并保留管理员专属入口。

### 6.3 测试结果分析
从当前验证结果看，系统主流程已经能够运行，且之前最明显的角色边界问题已经得到修复，管理员、教师和助教在页面访问和操作能力上形成了相对清晰的区分。这说明系统已经具备毕业设计演示所需的稳定基础。

但测试也反映出当前版本仍存在一定局限。首先，学生搜索、预警筛选和分页仍以基础实现为主，尚未形成统一的后端查询规范。其次，预警生成当前仍按课程全量扫描，后续在数据量扩大时需要考虑缓存、增量刷新或异步任务。再次，系统虽然已经完成期末成绩独立建模，但围绕期末成绩的统计查询和更细粒度图表展示仍有继续优化空间。上述问题不会阻断系统演示，但表明当前版本仍处于可演示、可验证而非完全成熟的阶段。

因此，本文对系统状态的表述应以“完成主流程与基础权限控制，并建立了最小回归验证机制”为宜，而不应夸大为“已达到生产级完善状态”。

## 第七章 结论

### 7.1 工作总结
本文围绕高校过程性教学监督需求，完成了一套基于 Python 的教学效果监督系统的分析、设计与实现。系统以 Flask 为后端基础框架，以 MySQL 为默认数据库，以 Pandas 和 OpenPyXL 处理教学表格数据，以原生 JavaScript、Bootstrap 和 Chart.js 实现前端页面和图表展示，构建了从登录认证、角色控制、基础数据管理到数据导入、综合预警、处理反馈和 Excel 导出的主要业务闭环。

从当前实现效果来看，系统已经在以下方面形成成果：一是实现了管理员、教师、助教三角色的页面与能力区分，管理员独立用户管理界面已恢复；二是实现了课程、班级、学生及多类教学数据的维护与导入，其中期末成绩已经形成独立模型、独立导入和独立导出链路；三是实现了综合预警生成与处理链路；四是建立了自动化回归验证机制。这些成果表明，系统已具备较为完整的毕业设计展示价值。

### 7.2 不足与展望
尽管当前版本已经完成主流程，但仍存在一些需要进一步完善的方面。

第一，学生搜索、预警筛选和统计查询仍需继续后端化，以提升大数据量场景下的可维护性。第二，预警生成目前采用全量扫描机制，后续可考虑引入缓存、异步任务或增量计算，以提升系统在更大规模场景下的可用性。第三，围绕期末成绩的图表呈现和统计维度还可以继续扩展，以便提供更细粒度的教学分析。第四，自动化测试虽然已经覆盖核心业务闭环，但仍可继续扩展到更多导入模板与复杂边界场景。

后续工作可从以下方向推进：统一后端查询参数规范；优化预警生成性能；扩展期末成绩相关统计展示；继续补充导入、导出、课程管理和异常数据场景的自动化测试。
