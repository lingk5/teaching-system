# 基于Python的教学效果监督系统的设计与实现

## 摘要
随着高校教学活动逐步数字化，课堂考勤、作业提交、随堂测验和课堂互动等过程性数据不断积累。相比期末成绩这类结果性数据，过程性数据更能反映学生在学习过程中的真实状态，但在实际教学中，这些数据往往分散在不同表格、不同平台和不同教师手中，统计方式不统一、处理效率较低，导致教师难以及时发现学业风险学生，也难以形成持续性的干预依据。针对这一问题，本文设计并实现了一套基于 Python 的教学效果监督系统。

系统采用前后端分离架构。后端以 Flask 为核心框架，结合 SQLAlchemy 完成用户、课程、班级、学生、过程数据和预警记录等业务对象建模；数据库当前默认使用 MySQL，并通过 ORM 保持后续迁移的灵活性；在数据处理环节，引入 Pandas 与 OpenPyXL 完成 Excel 数据导入、字段清洗、格式校验和错误反馈；前端采用原生 HTML、CSS 与 JavaScript 模块化开发，结合 Bootstrap 构建页面布局，结合 Chart.js 实现学情信息的图表展示。系统围绕教学监督场景实现了登录认证、角色权限控制、管理员用户管理、课程与学生管理、多类型教学数据导入、综合预警生成与处理、基础统计与报表导出等功能。

在预警机制方面，系统基于出勤、作业、测验和互动四类指标构建综合评分模型，并结合等级划分规则生成黄色、橙色和红色预警，辅助教师快速识别重点关注对象。与只依赖单一阈值判断的做法相比，当前实现更强调多维度汇总和结果解释，能够为教师提供较为直观的原因说明和处理建议。测试结果表明，系统当前版本已经完成主流程闭环和三角色基础权限分离，能够满足毕业设计演示与论文论证的需要。

**关键词**：教学监督；Python；Flask；数据导入；学情预警；角色权限控制

## ABSTRACT
With the digitalisation of higher education, process data such as attendance, homework, quizzes, and classroom interaction are generated continuously. Compared with final scores, process data can better reflect students' learning status during the semester. However, in actual teaching practice, these data are often scattered across spreadsheets and platforms, which leads to inconsistent statistical standards and low management efficiency. As a result, teachers find it difficult to identify at-risk students in time and carry out continuous intervention. To address this problem, this thesis designs and implements a teaching effectiveness monitoring system based on Python.

The system adopts a front-end and back-end separated architecture. The back end is built with Flask and SQLAlchemy to model users, courses, classes, students, process data, and warning records. MySQL is used as the default database in the current version, while the ORM layer preserves migration flexibility. Pandas and OpenPyXL are introduced to support Excel-based data import, field cleaning, format validation, and error feedback. The front end is developed with native HTML, CSS, and modular JavaScript, combined with Bootstrap for page layout and Chart.js for data visualisation. The system implements login authentication, role-based access control, administrator user management, course and student management, multi-type data import, comprehensive warning generation and handling, as well as basic statistics and report export.

For the warning mechanism, the system integrates attendance, homework, quiz, and interaction indicators into a comprehensive scoring model, and classifies students into yellow, orange, and red warning levels. Compared with single-threshold judgement, the current implementation emphasises multidimensional aggregation and interpretable results, which helps teachers understand the main reasons behind risks and take timely action. Current verification results show that the system has completed the core workflow and basic role separation among administrator, teacher, and assistant, making it suitable for graduation project demonstration and thesis discussion.

**Key words**: teaching supervision; Python; Flask; data import; early warning; role-based access control

## 第一章 绪论

### 1.1 研究背景
近年来，高校教学活动逐渐从单纯依赖结果评价转向更加重视过程评价。尤其是在混合式教学和信息化教学环境中，学生在整个学期中会持续产生大量过程性数据，例如签到记录、作业得分、随堂测验成绩、课堂互动次数和阶段性预警信息。这些数据如果能够被及时采集、清洗、分析并形成反馈，就可以帮助教师在期中阶段发现问题，而不是等到期末成绩出来后再进行被动补救。

但在实际教学管理中，过程数据常常分散在不同渠道之中：有的存在教师自己维护的 Excel 表格中，有的来自在线学习平台，有的来自课堂互动工具，还有一些依赖人工汇总。由于采集口径不统一、格式不规范、缺少持续的数据整理机制，教师往往需要投入大量时间进行重复统计，最终得到的结果也难以形成统一的学情视图。这种状况不仅影响教学效率，也削弱了过程性评价在教学改进中的实际作用。

在这一背景下，设计一套面向教学监督场景的轻量化系统具有现实意义。该系统不需要完全替代学校现有教务平台，而是作为过程性数据的聚合、分析和预警工具，帮助管理员维护基础数据，帮助教师完成过程信息录入和查看，也为助教提供只读辅助能力，从而形成更清晰的教学监督链路。

### 1.2 研究意义
本课题的研究意义主要体现在三个方面。

第一，具有明显的教学管理意义。系统将课程、班级、学生、考勤、作业、测验、互动和预警等信息集中到统一平台中，有助于减少教师在数据整理上的重复劳动，提高教学数据的可追踪性和可查询性。

第二，具有过程干预意义。与传统只关注期末成绩的做法相比，本系统强调过程性数据的持续汇总与预警生成。教师可以在学期中根据学生的综合得分和预警等级进行重点关注，从而把教学干预前移。

第三，具有工程实践意义。系统采用 Flask、SQLAlchemy、Pandas、MySQL、Bootstrap 和原生 JavaScript 等较为成熟且成本可控的技术栈，实现了一套结构清晰、便于部署和扩展的 Web 系统，为类似教学辅助平台的开发提供了可参考的实现路径。

### 1.3 国内外研究现状
从国内外研究情况来看，学习分析与教学监测已经成为教育信息化中的重要方向。国外相关系统通常依托较为完整的学习管理平台，对学习行为数据进行采集和建模，强调数据驱动教学决策。国内近年来也在智慧教育和教学评价改革的推动下逐步重视过程性数据的利用，部分平台已经能够提供考勤统计、成绩分析和课堂互动记录等功能。

但从中小规模教学场景出发，现有方案仍然存在若干不足。其一，很多平台功能较重，部署和定制成本较高，不适合快速落地到具体课程或院系场景。其二，不同来源的数据在实际使用中仍然需要教师自行整理，系统间的联通性较弱。其三，部分预警功能更偏向静态展示，对结果的解释性不足，教师虽然能够看到学生被标记为异常，却难以快速定位主要问题来源。

因此，开发一套以“轻量部署、角色清晰、导入友好、预警可解释”为特点的教学效果监督系统，既符合当前教学实际，也能为后续进一步扩展提供基础。

### 1.4 论文主要工作与结构安排
本文围绕教学效果监督系统的分析、设计与实现展开，主要完成了以下工作：

1. 设计了基于 Flask 的前后端分离系统架构，完成认证、课程、学生、数据导入、预警和导出等模块划分。
2. 实现了管理员、教师、助教三类角色的页面访问与能力控制，恢复了管理员独立用户管理界面。
3. 基于 MySQL 和 SQLAlchemy 建立了课程、班级、学生、教学过程数据和预警记录等核心数据模型。
4. 基于 Pandas 和 OpenPyXL 实现了多类 Excel 数据导入、清洗和错误反馈机制。
5. 实现了综合预警引擎，对学生学习状态进行分级预警，并提供基础原因说明与处理建议。

## 第二章 相关平台及关键技术

### 2.1 Python 与 Flask 框架
Python 具有语法清晰、生态丰富、开发效率较高等特点，尤其适合数据处理与 Web 服务结合的场景。本系统以后端接口服务和数据处理为核心，因此选择 Python 作为主要开发语言。

在 Web 框架方面，系统采用 Flask 作为后端基础框架。Flask 本身结构轻量、路由机制清晰，适合在毕业设计场景下快速构建 RESTful API。当前系统通过蓝图机制将接口划分为认证、课程管理、数据导入、学情分析、预警处理和数据导出等模块，既降低了代码耦合度，也便于后续迭代。

在身份认证方面，系统采用基于 JWT 的无状态鉴权机制。登录成功后，后端会将用户标识和角色信息写入令牌，并在后续请求中完成身份校验。相比传统服务端会话方案，该方式更适合前后端分离的应用结构。

### 2.2 SQLAlchemy、MySQL 与数据建模
数据库是系统稳定运行的基础。当前项目默认使用 MySQL 作为关系型数据库，主要原因在于其部署成熟、使用广泛，并能够满足教学监督系统对课程、学生和过程数据的持久化需求。

为了降低业务代码与底层 SQL 的耦合，系统使用 SQLAlchemy 作为 ORM 工具，将用户、课程、班级、学生、考勤、作业、测验、互动和预警等对象映射为数据库模型。通过这种方式，开发者可以在保持数据结构清晰的同时，提高查询、维护和扩展效率。

当前系统中，用户与课程之间是一对多关系，课程与班级之间是一对多关系，班级与学生之间是一对多关系，学生与多类过程数据之间也是一对多关系。预警记录与学生关联，用于存储当前版本的风险等级、原因说明和处理状态。该建模方式能够较好支撑当前业务流程。

### 2.3 Pandas、OpenPyXL 与数据导入
教学监督系统的一个典型场景是教师批量导入 Excel 数据。由于教师实际维护的表格格式常常存在空行、空格、字段缺失、状态值不规范等问题，系统必须具备较强的数据容错能力。

Pandas 在表格数据读取、缺失值处理和字段标准化方面具有较高效率，OpenPyXL 则适合处理 Excel 文件结构和导出样式。本系统在导入环节结合这两类工具，对学生名单、考勤、作业、测验和互动数据进行解析、校验和落库，并将成功条数、失败条数和错误原因返回给前端。这一机制是系统实用性的重要基础。

### 2.4 前端技术与可视化实现
当前前端采用原生 HTML、CSS 和 JavaScript 模块化开发，没有引入 Vue、React 等重型框架。这种实现方式虽然在组件化能力上不如大型前端框架完整，但更符合当前项目轻量、直接和易部署的目标。

页面层面使用 Bootstrap 完成布局、表单和表格样式，以减少基础界面开发成本；交互层面通过 `request.js` 封装网络请求，通过 `auth.js` 统一处理登录状态、角色主页跳转、页面访问控制和基于能力的控件显示；图表层面使用 Chart.js 绘制预警分布、趋势图和雷达图等可视化内容。虽然当前部分页面仍保留模拟数据兜底逻辑，但整体交互骨架已经形成。

## 第三章 系统需求分析

### 3.1 可行性分析
#### 3.1.1 技术可行性
从技术角度看，系统所采用的 Flask、SQLAlchemy、MySQL、Pandas、Bootstrap 和 Chart.js 都属于成熟技术，社区资料丰富、实现路径清晰，适合毕业设计项目的开发与维护。当前仓库已经完成主流程代码组织，并具备基础测试，说明该技术路线具备落地可行性。

#### 3.1.2 经济可行性
系统采用的核心框架和工具大多为开源方案，不需要额外的商业授权成本。对于教学演示场景而言，一台普通开发环境即可完成运行、展示和简单测试，部署成本较低，符合毕业设计项目的资源条件。

#### 3.1.3 业务可行性
从教学业务出发，课程过程监督确实存在基础数据分散、统计负担重和干预不及时的问题。当前系统以课程、班级和学生为对象，围绕教师日常管理流程设计，能够覆盖登录、维护、导入、预警、处理和导出等关键环节，业务目标明确，场景具有可行性。

### 3.2 角色与功能需求分析
当前系统面向三类角色开展设计，分别为管理员、教师和助教。

#### 3.2.1 管理员需求
管理员负责系统级基础维护，主要需求包括：

1. 登录系统并进入管理员控制台。
2. 创建、查看、修改教师和助教账号。
3. 查看课程、学生和整体统计信息。
4. 参与课程维护、数据导入和预警查看。
5. 在需要时处理预警与导出基础报表。

#### 3.2.2 教师需求
教师是系统主要使用者，主要需求包括：

1. 登录系统并进入教师仪表盘。
2. 创建课程、维护班级和学生基础数据。
3. 导入考勤、作业、测验、互动等过程数据。
4. 生成和查看学生预警，记录处理情况。
5. 查看学情统计信息并导出学生名单、成绩、考勤和预警报表。

#### 3.2.3 助教需求
助教当前承担辅助查看和基础支持任务，主要需求包括：

1. 登录系统并查看课程、学生、学情分析和预警页面。
2. 辅助教师了解当前班级学情状态。
3. 不具备用户管理、课程创建、数据导入和预警处理等关键写操作权限。

这种角色划分能够体现教学管理中的职责差异，也与当前系统实际实现保持一致：管理员拥有全部能力，教师拥有课程和教学数据管理能力，助教为只读辅助角色。

### 3.3 功能需求分析
结合当前仓库实现，系统功能需求可概括为以下几个部分。

#### 3.3.1 登录认证与权限控制
系统需要支持登录、身份验证、角色识别和页面访问控制。不同角色登录后应进入不同首页，并看到与自身职责相符的功能入口。管理员需要拥有独立用户管理页面，教师不应拥有用户管理能力，助教不应拥有课程创建和导入能力。

#### 3.3.2 基础数据管理
系统需要支持课程、班级和学生信息维护，能够建立课程到班级、班级到学生的层级关系。管理员和教师需要具备相应的数据维护能力，助教仅保留查看权限。

#### 3.3.3 教学数据导入
系统需要支持通过 Excel 批量导入学生名单、考勤、作业、测验和互动数据，并在导入失败时返回具体错误原因，帮助教师快速修正数据源。

#### 3.3.4 预警与处理
系统需要对学生的过程性数据进行计算，并生成可视化的风险提示与处理建议。教师和管理员应能够查看并处理预警，助教仅能查看。

#### 3.3.5 数据展示与导出
系统需要提供基础统计与报表导出能力，方便教师查看课程情况，并将学生名单、成绩、考勤和预警数据导出为 Excel 文件。

### 3.4 非功能需求分析
#### 3.4.1 易用性
系统面向教师和教学管理人员使用，因此界面应尽量简洁，操作路径应保持直接。当前版本通过固定侧边栏、统一表格和模态框表单组织主要功能，降低了学习成本。

#### 3.4.2 可维护性
后端通过蓝图、服务层和工具函数拆分模块，前端通过脚本模块和页面职责分离来降低耦合，能够提升后续维护效率。数据库模型与接口文件职责相对明确，也有利于版本迭代。

#### 3.4.3 安全性
系统当前采用 JWT 鉴权与角色能力矩阵控制关键操作，在用户管理、课程创建、数据导入和预警处理等环节增加了权限校验，能够为基础教学场景提供必要的安全边界。

## 第四章 系统设计

### 4.1 总体架构设计
系统采用前后端分离架构，整体可分为表现层、接口层、业务层和数据层四个部分。

表现层由浏览器中的前端页面构成，包括登录页、仪表盘、课程管理、学生管理、用户管理、数据导入、学情分析和预警中心等页面。这些页面通过统一的请求模块访问后端接口，并根据角色和能力动态控制菜单与按钮展示。

接口层由 Flask 蓝图构成，主要包括认证模块、课程模块、数据导入模块、分析模块、预警模块和导出模块。接口层负责接收请求、校验参数、调用业务逻辑，并向前端返回 JSON 数据或导出文件。

业务层主要包含权限判断、预警引擎、数据导入辅助函数和统计逻辑。预警引擎会根据学生最近一段时间的出勤、作业、测验和互动情况计算综合得分，并输出等级、原因和建议。

数据层由 MySQL 数据库和 SQLAlchemy 模型组成，负责持久化用户、课程、班级、学生和各类过程记录。系统总体架构如图 4.1 所示。

### 4.2 功能模块设计
#### 4.2.1 权限与页面映射设计
为了避免不同角色看到相同侧边栏和相同操作入口，系统在前后端同时建立了权限控制机制。前端 `auth.js` 中维护角色首页、可访问页面和能力集合，控制菜单显示、按钮裁剪和无权限页面跳转；后端 `permissions.py` 中维护统一能力矩阵，对关键接口进行二次校验。该设计能够降低只做前端隐藏而后端未拦截的风险。

#### 4.2.2 数据导入流程设计
数据导入流程围绕“上传文件、解析数据、校验字段、写入数据库、返回反馈”展开。用户在前端选择数据类型并上传 Excel 文件后，后端根据不同模板调用对应导入器，利用 Pandas 读取数据，再由辅助函数进行空值清理、字段格式标准化、合法性判断和错误收集。对于合法记录执行写入，对于异常记录保留错误信息并统一返回。该流程如图 4.3 所示。

#### 4.2.3 综合预警设计
系统当前以综合预警为核心场景。预警引擎会按课程读取学生列表，在最近分析时间范围内计算出勤分、作业分、测验分和互动分，再通过统一权重配置生成综合分数。随后根据阈值划分黄色、橙色和红色预警，并生成原因说明和建议文本。教师或管理员可以在预警中心继续记录处理结果。该流程如图 4.4 所示。

#### 4.2.4 导出设计
数据导出模块当前支持学生名单、成绩报表、考勤统计和预警报告导出，文件格式为 Excel。导出逻辑通过查询数据库后利用 Pandas 和 OpenPyXL 生成带基础样式的工作簿，方便教师直接留存或继续加工。

### 4.3 数据库设计
#### 4.3.1 核心实体设计
结合当前实现，系统涉及以下核心实体：

1. 用户实体，用于存储管理员、教师和助教账号。
2. 课程实体，用于存储课程基础信息并与教师关联。
3. 班级实体，用于表示课程下的班级结构。
4. 学生实体，用于表示学生基本资料并与班级关联。
5. 出勤、作业、测验和互动实体，用于存储过程性教学数据。
6. 预警实体，用于存储学生预警等级、原因说明、建议和处理状态。

#### 4.3.2 实体关系说明
当前数据库关系如下：

1. 一个用户可以关联多门课程。
2. 一门课程可以包含多个班级。
3. 一个班级可以包含多个学生。
4. 一个学生可以拥有多条出勤记录、作业记录、测验记录和互动记录。
5. 一个学生在不同时间可以产生多条预警记录。

该设计既符合当前业务使用方式，也能支撑导入、预警和导出等主要模块。数据库 E-R 图可按图 4.5 绘制。

## 第五章 系统实现

### 5.1 开发环境与项目组织
当前系统采用 Python 3 环境运行，后端主要依赖 Flask、Flask-JWT-Extended、Flask-SQLAlchemy、Pandas、OpenPyXL 和 PyMySQL 等库；前端采用浏览器静态页面模式，主要依赖 Bootstrap 与 Chart.js。项目通过 `create_app` 应用工厂初始化数据库、JWT 和跨域配置，并在启动时创建默认演示账号 `admin`、`teacher` 和 `assistant`，便于本地演示与测试。

在项目组织上，后端路由位于 `backend/app/routes`，模型位于 `backend/app/models`，预警与权重逻辑位于 `backend/app/services`；前端页面位于 `frontend/src/pages`，公共脚本位于 `frontend/src/js`。这种组织方式能够使论文中提到的业务模块与代码结构一一对应。

### 5.2 登录认证与角色权限实现
系统的登录逻辑位于 `auth.py` 和 `login.html`。用户提交用户名和密码后，后端校验账号状态并生成携带角色信息的 JWT 令牌，前端将令牌和用户信息写入本地存储。此后所有受保护页面都会先执行认证检查，若令牌缺失或过期则自动跳转至登录页。

角色权限控制由前端和后端共同完成。前端 `auth.js` 维护三类角色的首页映射：管理员默认进入 `users.html`，教师进入 `dashboard.html`，助教进入 `analytics.html`。同时，前端还根据能力集合控制“用户管理”“创建课程”“数据导入”“生成预警”“处理预警”等入口的显示。后端 `permissions.py` 再对关键接口进行校验，避免出现仅靠界面隐藏而导致的越权访问。

这一实现使系统当前已经摆脱了管理员、教师和助教侧边栏完全相同、权限边界不清的问题。管理员拥有完整能力，教师不具备用户管理能力，助教不具备课程创建、数据导入和预警处理能力。

### 5.3 管理员用户管理与基础数据维护实现
为满足管理员与教师界面分离的要求，系统恢复并实现了管理员专用的 `users.html` 页面。该页面仅允许管理员访问，并对接后端 `/api/auth/users` 相关接口，实现用户列表查看、创建用户和编辑用户等操作。管理员可以在系统中维护教师和助教账号，也可以根据需要停用账号。

在课程与学生数据维护方面，后端 `courses.py` 负责课程、班级和学生的查询与增删改查逻辑，前端分别通过课程管理页和学生管理页展示层级信息。课程页面支持查看课程下的班级情况和预警人数，学生页面支持查看学生列表及基础画像信息，构成了教学监督所需的基础数据闭环。

### 5.4 数据导入实现
数据导入模块是本系统的重要实现内容之一。当前后端 `data.py` 支持学生、考勤、作业、测验、期末成绩复用接口和互动等多类数据导入。系统会根据数据类型调用对应导入器，对表格内容进行字段验证、空值处理、状态转换和错误记录。

前端数据导入页面为教师和管理员提供模板下载、文件上传和结果反馈功能。导入结束后，系统会返回成功数量、失败数量及具体错误原因，帮助用户快速定位问题。这一实现较好地契合了教师使用 Excel 维护过程数据的实际习惯。

### 5.5 预警生成、处理与可视化实现
预警生成逻辑由 `warning_engine.py` 完成。引擎当前选取出勤、作业、测验和互动四类指标，分别换算为百分制后参与综合评分。评分结果由统一权重配置管理，并进一步映射为黄色、橙色和红色三个等级。对于达到预警条件的学生，系统会生成原因说明和建议文本，并在预警记录表中保存。

预警中心页面负责展示预警信息及处理状态。教师和管理员可以在该页面查看学生风险等级、处理建议和处理进展，并记录预警处理结果。助教仅保留查看权限，不允许进行处理操作。

在数据可视化方面，系统当前提供仪表盘和学情分析页面，用于展示统计信息和图表内容。其中预警分布、学生信息展示和部分图表已经能够形成基本展示效果，但当前版本的仪表盘和分析页仍保留部分模拟数据兜底逻辑，因此这些页面更适合作为演示型分析界面，而不是完全成熟的统计平台。

### 5.6 数据导出实现
系统当前已经实现基础 Excel 导出能力。后端 `export.py` 支持导出学生名单、成绩报表、考勤统计和预警报告，并通过 OpenPyXL 为表头和边框添加基础样式。对于教师而言，这一功能便于将系统内的数据继续用于教学留档或线下分析。

需要说明的是，当前实现侧重 Excel 报表输出，并未实现论文二稿中曾经提到的 PDF 学情报告能力。因此在论文正文中应以 Excel 导出为准。

## 第六章 系统测试

### 6.1 测试环境与测试方法
为了验证当前版本是否具备演示与论文论证基础，系统采用了自动化回归验证与源码级检查相结合的方式进行测试。后端使用 Python `unittest` 对角色权限和页面回归进行验证，前端使用 `node:test` 对角色矩阵和控件裁剪逻辑进行验证，同时结合 `compileall` 和补丁检查命令确认代码基础状态。

本次测试围绕以下几个重点展开：

1. 管理员、教师、助教三角色的权限是否已经分离。
2. 管理员用户管理界面与接口是否可用。
3. 助教是否仍然可以进行不应拥有的高权限操作。
4. 前端页面访问与按钮显示是否与后端能力控制保持一致。
5. 当前代码是否能够通过基础自动化回归和编译检查。

### 6.2 测试结果
根据当前代码基线的实际验证结果，系统已经建立最小自动化回归测试集。后端共完成 11 条测试，前端共完成 3 条测试，此外通过了 Python 编译检查和补丁格式检查。

在角色权限方面，测试确认了以下结论：

1. 管理员登录后默认进入用户管理页面。
2. 管理员可以访问用户列表接口。
3. 教师访问用户管理接口会被拒绝。
4. 助教无法导入课程数据。
5. 助教无法修改学生信息。
6. 助教无法处理预警。
7. 管理员页面文件已存在且已接真实接口，不是空白演示页。

在前端回归方面，测试确认 `auth.js` 中的角色主页、页面访问矩阵和能力裁剪逻辑能够按预期工作，能够正确隐藏助教不应看到的高权限控件，并保留管理员专属入口。

### 6.3 测试结果分析
从当前验证结果看，系统主流程已经能够运行，且之前最明显的角色边界问题已经得到修复，管理员、教师和助教在页面访问和操作能力上形成了相对清晰的区分。这说明系统已经具备毕业设计演示所需的稳定基础。

但测试也反映出当前版本仍存在一定局限。首先，仪表盘和学情分析页面仍包含模拟数据兜底逻辑，统计展示尚未完全由真实接口驱动。其次，教师对本人课程之外的数据访问边界还未彻底收口。再次，导出模块与预警引擎之间的权重口径仍需继续统一。上述问题不会阻断系统演示，但表明当前版本仍处于可演示、可验证而非完全成熟的阶段。

因此，本文对系统状态的表述应以“完成主流程与基础权限控制，并建立了最小回归验证机制”为宜，而不应夸大为“已达到生产级完善状态”。

## 第七章 结论

### 7.1 工作总结
本文围绕高校过程性教学监督需求，完成了一套基于 Python 的教学效果监督系统的分析、设计与实现。系统以 Flask 为后端基础框架，以 MySQL 为默认数据库，以 Pandas 和 OpenPyXL 处理教学表格数据，以原生 JavaScript、Bootstrap 和 Chart.js 实现前端页面和图表展示，构建了从登录认证、角色控制、基础数据管理到数据导入、综合预警、处理反馈和 Excel 导出的主要业务闭环。

从当前实现效果来看，系统已经在以下方面形成成果：一是实现了管理员、教师、助教三角色的页面与能力区分，管理员独立用户管理界面已恢复；二是实现了课程、班级、学生及多类过程数据的维护与导入；三是实现了综合预警生成与处理链路；四是建立了最小自动化回归验证机制。这些成果表明，系统已具备较为完整的毕业设计展示价值。

### 7.2 不足与展望
尽管当前版本已经完成主流程，但仍存在一些需要进一步完善的方面。

第一，学情分析页面和仪表盘中的部分图表仍保留模拟数据兜底逻辑，真实统计展示能力还需继续加强。第二，教师资源所有权边界尚未完全收口，后续应进一步限制教师只访问本人课程、班级和学生数据。第三，导出模块与预警模块之间的评分口径仍需统一，以避免不同模块得出不同统计结果。第四，自动化测试当前仍以最小回归为主，后续应继续覆盖导入、导出和课程管理等完整链路。

后续工作可从以下方向推进：继续完成图表页真实数据接入；统一评分配置和统计口径；补充教师资源范围控制；扩展测试范围；在数据规模增加后引入缓存、异步任务或增量计算机制，以提升系统在更大规模场景下的可用性。
